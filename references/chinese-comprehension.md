# 中文深度理解协议

> 目标：让模型听懂更复杂、更地道、更"不按字面"的中文。
> 适用：DeepSeek-V4-Pro / Xiaomi-MiMo-V2.5-Pro。

---

## 1. 为什么需要专门处理中文

中文大量使用倒装、省略、成语、歇后语、双关、语气词、网络用语。
逐字直译往往丢失真实意图。本协议在"字面 → 意图 → 目标"理解链
（见 dialogue-optimization.md）之上，专门补齐中文特有现象的处理。

---

## 2. 词语倒装与语序还原

中文口语常打乱标准语序，先还原为"主谓宾"再理解：

| 现象 | 例子 | 还原 |
|---|---|---|
| 谓语前置/倒装 | "出来吧，那个文件" | "把那个文件（打开/拿）出来" |
| 宾语前置 | "这个 bug 我搞不定" | "我搞不定这个 bug" |
| 状语后置 | "改一下代码，用 Python" | "用 Python 改一下代码" |
| 主语后置 | "挺好用的，这个工具" | "这个工具挺好用" |
| 强调倒装 | "重要的是先跑通" | 重点 = "先跑通" |

处理步骤：① 识别成分错位 → ② 在内部重排为标准语序 → ③ 按重排后的句子理解。

---

## 3. 成语 / 俗语 / 歇后语

- **成语**：理解其**比喻义**而非字面。如"画蛇添足"= 做了多余的事，
  "雪上加霜"= 让坏情况更糟，"对症下药"= 针对根因解决。
- **俗语/谚语**："磨刀不误砍柴工"= 准备工作值得做；"换汤不换药"= 没有实质改变。
- **歇后语**：取后半解。如"竹篮打水——一场空"= 白费力气。
- 用户用成语描述需求时，把比喻义翻译成**具体技术动作**再执行。
  例："这段代码牵一发而动全身" → 含义：耦合度高，改动影响面大 → 行动：先评估依赖再动。

---

## 4. 省略、指代与语气

- **省略主语/宾语**：中文高频省略，按上下文补全。"删了吧"→ 删什么？回指最近提到的对象。
- **指代**："这个""那个""它""刚才那段"——绑定到最近、最相关的指称对象，不确定就问。
- **语气词**：吧/啊/呢/嘛/哈/呗 传递态度（试探、确认、不耐烦、随意），
  用于判断**确定性**和**紧急度**，但不改变任务本身。
- **反问 / 反话**："这能行？"= 表示怀疑；"你倒是快点啊"= 催促。识别真实情绪。

---

## 5. 网络用语 / 缩写 / 口语

- 常见缩写：yyds（永远的神/很强）、xswl（笑死我了）、u1s1（有一说一/说实话）、
  dbq（对不起）、awsl 等——理解情绪色彩，不影响任务执行的就略过。
- 口语化技术词："跑一下"=运行，"整一个"=做一个，"搞定"=完成，
  "崩了"=崩溃/报错，"卡住了"=阻塞/无响应，"报红"=报错。
- 不确定的新词/黑话：先按最可能含义推进，必要时一句话确认。

---

## 6. 双关、多义与歧义消解

- 一词多义按**领域上下文**消解。如"内存泄漏"中的"泄漏"≠ 信息泄漏。
- 中英混用：用户常中英夹杂（"这个 function 有 bug"），正常理解，不必纠正风格。
- 真歧义且影响结果 → 列出候选理解，选最合理一种推进，分歧大才发问。

---

## 7. 中文理解自检

- [ ] 有没有语序错位需要先还原？
- [ ] 成语/俗语我取的是比喻义而非字面？
- [ ] 省略的主语/宾语/指代补对了吗？
- [ ] 语气/反话背后的真实情绪和确定性判断对吗？
- [ ] 把"地道中文"翻译成"具体技术动作"了吗？
