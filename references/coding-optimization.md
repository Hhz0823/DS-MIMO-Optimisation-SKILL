# 代码编程优化协议

> 目标：写出可读、可验证、安全、最小充分的代码。
> 适用：DeepSeek-V4-Pro / Xiaomi-MiMo-V2.5-Pro，宿主 Claude Code / Codex。

---

## 1. 先读后写（Read Before Write）

- 动手前读相关现有代码：理解项目的结构、风格、约定、依赖。
- 看 `package.json` / `pom.xml` / `Cargo.toml` / `Makefile` 等配置，确认构建、测试、lint 命令。
- 不引入项目里没有的新库/新模式，除非有充分理由并说明。
- 写出的代码要"读起来像周围的代码"：同命名风格、同注释密度、同惯用法。

---

## 2. 最小充分（Minimal & Sufficient）

- 只解决被问到的问题。
- 不过度设计、不加多余抽象、不写超出需求的防御性代码。
- 修 bug 不顺手"重构"周边；加简单功能不强行加可配置性。
- 但较大功能该改的现有设计要改，保证整体连贯正确。

---

## 3. 安全默认（Secure by Default）

- 数据库：参数化查询，杜绝 SQL 注入。
- 输入：校验与清洗。
- 错误：正确处理，不吞异常、不泄露敏感信息。
- 依赖：用固定/锁定版本；警惕仿冒包名（typosquatting）。
- Shell：对用户输入正确转义/引用，优先数组式传参而非字符串拼接。
- **网络端点**：创建对外 API/服务时，若缺少鉴权/访问控制，**主动提示**安全风险，即使用户没问。
- 密钥：不回显 `.env`、私钥、token 等敏感值；用 key 名引用而非值。

---

## 4. 可验证（Verify）

- 改完代码 → 跑构建/编译。
- 构建不自动跑测试 → 单独跑相关测试。
- 新功能/修 bug → 写并跑测试；没有测试框架就按该语言生态的标准选型搭一个。
- 跑不了（缺依赖/环境受限）→ 明确说明原因，不假装通过。
- 验证发现的错误，先修好再交付。
- 清理验证过程产生的临时文件。

---

## 5. 可读性与文档

- 复杂逻辑加注释解释"为什么"，而非"做了什么"。
- 命名清晰、表意。
- 提供完整可运行的示例（在合适时）。
- 保证生成代码符合无障碍（accessibility）要求（涉及 UI 时）。

---

## 6. 编程自检清单

- [ ] 读过相关现有代码、匹配了项目风格？
- [ ] 只解决了被问到的问题，没过度设计？
- [ ] 用了安全默认（注入防护、输入校验、错误处理）？
- [ ] 暴露端点是否提示了鉴权？
- [ ] 跑过构建与测试？跑不了是否说明了原因？
- [ ] 清理了临时文件？
