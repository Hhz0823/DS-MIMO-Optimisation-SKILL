# 对话联想与补全协议

> 目标：当用户的话不完整、缺主语、缺要求、意思没说全时，主动联想补全；
> 找出真正的主语与要求，把"没说出口但显然想要"的内容补齐，再去执行。
> 适用：DeepSeek-V4-Pro / Xiaomi-MiMo-V2.5-Pro。

---

## 1. 何时启用联想补全

用户的输入出现以下任一情况时启用：

- **缺主语**：不知道在说谁/什么。"删掉吧""跑一下""不行，再来"。
- **缺宾语/对象**：动作有了，对象没说。"优化一下""改成那样"。
- **缺要求/标准**：要做什么有了，但要做到什么程度没说。"快一点""好看点""规范点"。
- **意思没说全**：半句话、被打断、省略了关键条件。"如果报错……""那个接口的话"。
- **指代悬空**："这个""那个""它""上面那个"找不到明确指向。

---

## 2. 联想补全的四步法

### 第一步：找主语（Who / What）

- 回扫对话历史，找最近、最相关的实体作为主语/对象。
- 优先级：当前句明示 > 上一轮讨论的对象 > 整场对话的主线对象 > 当前打开的文件/任务。
- 例："删掉吧" → 回指上一句提到的"那个没用的临时函数" → 主语 = 该函数。

### 第二步：补内容（What's missing）

- 把省略的成分按语法和语境补全成完整句。
- 半句话推断后续：用户说"如果报错……" → 联想"……希望我加上错误处理/打印日志/重试"。
- 补全要基于**已有线索**，不是凭空发明；线索不足时进入第四步澄清。

### 第三步：联想要求（Implied requirements）

把"没说出口但显然想要"的要求补齐。常见隐含要求：

| 用户说 | 联想出的真实要求 |
|---|---|
| "快一点" | 性能优化：减少耗时/复杂度/IO；先定位瓶颈 |
| "规范点" | 符合项目代码风格、命名约定、加注释、过 lint |
| "好看点" | UI：排版、对齐、间距、配色、响应式、无障碍 |
| "简单点" | 降低复杂度、减少依赖、更易读 |
| "稳一点" | 加错误处理、边界校验、测试、重试/回滚 |
| "做个登录" | 含校验、错误提示、安全（哈希/防注入）、状态管理 |
| "整理一下" | 去重、排序、分组、统一格式 |

> 联想出的隐含要求，执行前**简要说明**："我按'快'理解为优化性能、先定位瓶颈，
> 同时保持功能不变——如果你指的是别的，告诉我。"

### 第四步：验证或澄清

- 补全后，用一句话回述"我把你的意思补全为：X"，让用户有机会纠正。
- **能合理补全就补全并说明**（不打断节奏）；
- **补全的分歧会显著改变结果**（代价高、不可逆）才发问，且带选项："你是要 A 还是 B？"

---

## 3. 联想的安全边界

- **补内容，不改诉求**：联想是把用户的意思补完整，不是替他改成你想做的。
- **基于线索，不凭空捏造**：所有补全都要能在对话/代码/上下文里找到依据。
- **高风险不自动补**：删除、覆盖、上线、改权限等不可逆操作，缺关键信息时**必须**确认，不靠联想。
- **标注假设**：把联想出的部分标清楚，让用户一眼看出哪些是他说的、哪些是我补的。

## 4. 与其他协议的关系

- 先经 `chinese-comprehension.md` 还原倒装/成语/省略的字面，再用本协议做语义补全。

- 与 `dialogue-optimization.md` 的"纠正"配合：纠正是改错的，联想是补缺的，二者常同时发生。

## 5. 联想补全自检

- [ ] 缺的主语/对象找到了吗？绑定到对话里的正确实体了吗？
- [ ] 省略的内容、半句话的后续补全了吗？依据是否来自上下文？
- [ ] 没说出口的隐含要求联想到了吗？执行前说明了吗？
- [ ] 补全是"补意思"而非"改诉求"？
- [ ] 高风险操作缺信息时，是否改为确认而非自动补？
