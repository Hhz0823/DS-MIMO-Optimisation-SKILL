# 效率优化协议：缓存命中率与上下文使用率

> 目标：让对话更省 token、更快、更便宜——提高 prompt 缓存命中率，
> 提高上下文窗口的有效使用率，同时不损失理解与质量。
> 适用：DeepSeek-V4-Pro / Xiaomi-MiMo-V2.5-Pro，宿主 Claude Code / Codex。

---

## 1. 缓存命中率优化（Prompt Cache Hit Rate）

大模型推理通常对"前缀相同"的内容做 KV 缓存：**前缀不变 → 命中缓存 → 更快更省**。
一旦前缀的任意一处变了，**从变动点往后全部失效**。据此优化：

### 1.1 保持前缀稳定（最重要）

- 把**最稳定**的内容放最前面：系统指令、本 SKILL 协议、长期约定、项目背景。
- 把**最易变**的内容放最后面：当前问题、临时数据、本轮输入。
- 顺序固定：系统/协议 → 工具定义 → 历史对话 → 当前输入。不要每轮重排前面的块。

### 1.2 不要改写已固定的内容

- 已写入上下文的协议、文档、代码块，**不要为了"优化措辞"反复重写**——
  每改一个字，缓存就从那里断掉。要改就集中一次改完。
- 引用稳定文件用"路径 + 摘要"，而不是反复粘贴全文。

### 1.3 追加而非插入

- 新信息**追加到末尾**，不要插到中间——插入会让插入点之后的缓存全部失效。
- 修订靠"在末尾补充更正"，而非回头编辑前文（除非确有必要）。

### 1.4 注意缓存的时间窗口（TTL）

- prompt 缓存通常有有效期（常见约 5 分钟）。超时后即便前缀相同也需重算。
- 含义：**连续对话比间隔很久的对话更省**。长时间空闲后第一条会变慢变贵，属正常。
- 实践：相关的多个问题集中问完，比零散间隔问更高效。

### 1.5 稳定化检查清单

- [ ] 稳定内容在前、易变内容在后了吗？
- [ ] 有没有不必要地重写/重排已固定的前文？
- [ ] 新内容是追加而非插入吗？
- [ ] 是否在缓存窗口内连续推进，减少冷启动？

### 1.6 进阶命中率技巧

- **工具/函数定义固定化**：工具列表、schema、system 段一旦确定就不要动顺序或措辞；
  即便临时不用某工具，也不要从前缀里删——删了会让其后全部缓存失效。
- **变量后置、模板前置**：把"模板/骨架"放前（稳定），把"填进去的值"放后（易变）。
  例如代码审查协议固定在前，待审代码贴在最后。
- **批量优于零散**：同一前缀下连续追问多个相关问题，每问只新增末尾增量，全程命中前缀缓存。
- **避免时间戳/随机串污染前缀**：不要在靠前位置塞当前时间、UUID、计数器等每轮都变的内容，
  否则前缀每轮失效。需要这类信息就放到末尾。
- **稳定的分隔与格式**：标题层级、分隔线、字段顺序保持一致；格式抖动也会降低命中率。
- **复用而非重述**：前文已给过的背景/约定，用"如前所述"指代，不整段复制——
  既省 token，又不制造新的可变前缀。

### 1.7 命中率与正确性的边界

- 护缓存**不得**以牺牲正确性为代价：前文确有错误或已过时，**该改就改**，
  只是要"集中一次改完"，而不是零敲碎打反复改导致缓存反复失效。
- 一次性大改一次（缓存重建一次）优于多次小改（缓存断多次）。

## 2. 上下文使用率优化（Context Window Utilization）

上下文窗口有限且越长越慢越贵。目标是**用最少 token 装下最有用的信息**。

### 2.1 信息密度优先

- 表达高密度：去掉客套、重复、填充词，一句话能说清不用三句。
- 数据结构化：用表格/要点/键值，比大段散文更省 token 且更易检索。
- 引用代替粘贴：大文件给路径 + 关键片段 + 摘要，按需再读，不整段塞入。

### 2.2 分层加载（Progressive Disclosure）

- 主入口（SKILL.md）只放摘要，细则拆到 references，**用到才读**。
- 不预加载所有细节；按当前任务相关性选择性读取。
- 这正是本 SKILL 的目录设计：摘要常驻，细则按需展开。

### 2.3 及时清理与压缩

- 已完成、不再相关的中间产物（临时输出、探索过程、失败尝试），可在摘要中折叠为一行结论。
- 长输出（日志、构建结果）提炼关键行，不全量保留。
- 大任务用"状态快照"承载进度（见 context-memory.md），而非依赖完整历史。

### 2.4 输出也要省

- 回答篇幅与任务匹配：简单问题简短答，不堆标题和套话。
- 不复述用户已知的上下文；不逐条重复已经说过的内容。
- 代码改动只展示关键差异，不必整文件回贴（除非用户要）。

### 2.5 与记忆强化协同

- 上下文使用率优化 = "装得精"，记忆强化 = "丢了能找回"。
- 二者配合：压缩前提炼快照（省空间），压缩后重读快照（保记忆）。
- 详见 `context-memory.md`。

### 2.6 上下文使用率检查清单

- [ ] 信息是否高密度、结构化、无冗余？
- [ ] 大文件是引用+摘要，还是被整段塞入？
- [ ] 细则是否按需加载，而非全部预载？
- [ ] 失败尝试/中间过程是否折叠成结论？
- [ ] 输出篇幅是否与任务匹配，没有复述已知信息？

---

## 3. 缓存与上下文的权衡

- 二者有时冲突：为省上下文而频繁改写前文，会破坏缓存。**优先保前缀稳定**，
  清理只针对"末尾的、确实无用的"内容。
- 原则：**前文只追加不改写（护缓存），末尾按需精简（提使用率）。**
